# AI 编程协作实战：提问的质量，决定了代码的质量

> 同样的 AI 编程助手，为什么有人用出 10 倍效率，有人却觉得鸡肋？差距不在工具，在你的提问方式。

过去半年，我们在一个大型 A/B 实验平台的研发中深度使用了 AI Coding Agent。从最初「帮我写个方法」的随意指令，到后来形成一套结构化的协作方法，产出的代码质量差距在 3 倍以上。

这篇文章把踩过的坑和验证有效的经验整理出来，希望对你有用。

---

## 一、一个反直觉的认知

**Agent 的输出质量上限，不由模型版本决定，而由你给出的上下文决定。**

这不是玄学。想象你在带一个技术很强但完全不了解你项目的实习生——你只跟他说「把这个模块拆了」，他大概率会按自己的理解直接上手，方向偏了你要花几轮对话才能拽回来。

但如果你告诉他：「现在的问题是离线计算任务耦合在在线服务里，导致发布互相阻塞。要把它拆出去单独部署。注意，不能动对外 API，数据库连接配置要保持一套。你先分析现状，提一个分阶段方案，我确认后再执行。」

同样的实习生，同样你一句话——但他给你的东西质量完全不同。

## 二、提问四要素：一个简单的框架

经过几十次协作迭代，我们提炼出一个好 prompt 的四个要素：

```
1. 背景（1 句）  →  为什么要做这件事？
2. 约束（1-2 条）→  什么绝对不能动？
3. 期望输出      →  分析？方案？直接执行？先审再做？
4. 审查视角      →  用什么标准验证结果？
```

用一个真实场景对比：

| 维度 | 低信息量版本 | 高信息量版本 |
|------|------------|------------|
| 任务陈述 | 「把统计模块拆了」 | 「要把离线统计任务从在线服务拆出来单独部署」 |
| 约束 | 无 | 「不能动业务服务模块的 API；数据库连接配置保持一套」 |
| 期望输出 | 隐含（直接执行） | 「先分析现状，再给分阶段计划，我审完再执行」 |
| 审查视角 | 无 | 「执行完后，用『计算结果是否真正被写入』的视角再过一遍」 |

**低信息量版本的结果**：Agent 直接开始拆分，在方案设计阶段就可能与你的预期偏离，需要多轮修正。更糟的是，如果修正指令不及时，Agent 会沿着错误方向越走越远。

**高信息量版本的结果**：Agent 先探索代码结构、理解依赖关系、主动发现了一个架构反模式（统计模块既是库又是独立进程），再提出分阶段方案。你在动手前就完成了方向校准。

一个 prompt 多花 30 秒，省掉的是后续 30 分钟的纠错。

## 三、三种高杠杆的提问模式

### 模式一：探索式诊断——先理解，后决策

**你不需要知道答案，但你需要问对问题。**

不要上来就给操作指令。如果你隐约觉得哪里不对但又说不清，用「为什么」开头。

**真实案例**：有次我感觉服务依赖关系有点奇怪，就问了一句：「为什么业务服务模块会依赖统计计算模块？业务服务的定位是什么？」

这一问触发了 Agent 的架构分析，它跑了一遍代码后发现统计模块存在双模式反模式——同一个模块既要作为库被其他服务引用，又要作为独立进程部署。这两种模式的生命周期、配置、依赖方向完全不同，混在一起就是持续的技术债务。

如果我说的是「帮我把统计相关的引用理一下」，Agent 只会列一张表和几行引用计数，架构问题永远不会被暴露。

**适用场景**：你感到不对劲但说不清问题在哪——这正是 AI 诊断的强项。

### 模式二：外部审查 + 独立验证

团队协作中常见一个场景：同事或另一个工具给你提了一堆问题和建议。你会逐个核查再动手吗？还是会直接转发给执行者？

大多数人嘴上说前者，实际做的是后者。AI Agent 也一样——你直接把 16 个 issue 列表扔给它说「全改了」，它真的会逐项判断合理性吗？不会，它会当成待办清单无脑执行。

**正确做法**：「请独立检查核实这些审查结论。你认同的就修复，不认同的说明原因。」

我们做过一次对比：外部审查报告提了 16 个问题。如果不加核实指令，Agent 的修复采纳率是 100%（照单全收）。加了核实指令后，Agent 自己推翻了其中 4 个——有的因为审查者对业务逻辑理解有误，有的因为审查使用的上下文版本已经过时。

**关键话术**：「先核实，再执行。不同意的说明原因。」

### 模式三：审查时指定分析视角

不要只说「帮我审查一下这段代码」。如果你不告诉它看什么，它就看最表层的东西——命名、格式、明显的空指针。

**我们验证有效的审查视角**：

| 视角 | 适用场景 |
|------|---------|
| 「追踪每个计算结果的消费者——如果计算结果只被日志打印而未被任何字段使用，视为缺陷」 | 算法/数据管道 |
| 「检查枚举定义和数据字典是否同步」 | 状态机变更 |
| 「对连续型指标是否误用了离散分布的统计公式」 | 统计检验代码 |
| 「所有外部输入是否有空值和边界检查」 | 健壮性审查 |

**真实案例**：在统计管线审查中，我们要求 Agent 追踪方差缩减算法的每个计算结果：「这个函数输出了什么？谁消费了它？最终有没有写入报告？」

Agent 按这个视角跑完后发现：核心功能 `apply()` 的计算结果，只被一行 debug 日志消费，从未进入任何报告字段。

这个问题，编译检查发现不了，单测覆盖不了，功能测试也测不出来。只有「追踪数据流向最终消费」这个视角才能暴露。

同样的 Agent，同样的代码——换一个审查视角，发现的问题完全不同。

## 四、最常见的三个误区

### 误区一：把 Agent 当编译器

**典型症状**：「在 XX 类里加一个 YY 方法。」

你已经在替 Agent 做技术决策了。它只是帮你敲键盘。但你确定 YY 方法就是最优解吗？会不会有更简洁的方案你不知道？

**改进**：说目的，不说实现。把设计空间留给 Agent。

❌ 「给 UserService 加一个 getUserStats 方法」
✅ 「管理员面板需要看到用户的聚合统计数据。你看看现有代码结构，给个实现方案。」

### 误区二：一个 Prompt 里塞太多东西

**典型症状**：一段话里包含 5 个独立任务。

Agent 不是多线程 CPU，它按顺序处理信息。任务 A 的上下文会污染任务 B 的判断，任务 C 的错误会被带到任务 D。

**改进**：一个 prompt 聚焦一个主题。相关的子任务可以放一起，但独立主题必须拆开。多开几个会话窗口的成本，远低于在一个窗口里反复纠错。

### 误区三：执行完不给反馈信号

**典型症状**：Agent 执行完后你说「好的，继续下一步」。

对你来说「好的」是确认，对 Agent 来说是零信息——它不知道自己的输出是否符合你的预期，下次无法校准。

**改进**：给一个短反馈。

✅ 「拆分方案确认，结构和我预期一致，继续下一步。」
✅ 「调度模块我希望用独立的父 POM，不要继承统一父 POM，其他没问题。」

反馈不需要长，但需要在关键决策点上把方向校准一下。

## 五、三条经验规则

1. **复杂任务开头给硬约束。** 一两句话圈定禁区（不能改的 API、不能动的配置、不能引入的新依赖），能让 Agent 跳过大量无效探索，直接在你给的空间内找最优解。

2. **用「先分析现状 → 提方案 → 我审完再执行」控制节奏。** 这是投入产出比最高的话术。多花 1 分钟做方案对齐，可能节省 30 分钟的返工。

3. **外部审查结果必须要求独立核实。** 不同 Agent（或不同人）看到的信息视野不同。交叉验证不是不信任，是工程常识。没有独立核实的外部输入，不要直接作为执行指令。

4. **给结论的同时，给原因。** 「上次这么做有问题」没有信息量。要说「上次序贯检验对连续型收入数据误用了二项分布公式，导致方差为负数。这次碰到类似的统计检验代码，先确认分布假设是否正确。」

## 六、如何让下一次协作更好

每次和 AI Agent 协作完，花 2 分钟做三件事：

1. **沉淀决策和约束。** 把这次发现的非显而易见的坑、架构约束、命名约定写进项目知识库，下次 Agent 可以自动加载这些上下文。

2. **固化审查视角。** 把验证有效的审查维度（比如「追踪每个计算结果的最终消费者」）沉淀为团队代码审查清单。一个验证过的视角，可以反复复用。

3. **维护项目上下文配置。** 持续更新项目的模块角色定义、依赖策略、命名规范——Agent 每次启动都会自动加载，相当于每次都有「老员工」级别的上下文。

---

## 写在最后

AI 编程助手的进步速度很快，但有一个东西不会变：**工具的能力上限是你的表达上限**。你能多清晰地描述背景、约束和验收标准，它就能多好地帮你解决问题。

与其花时间研究各种 prompt 模板和技巧，不如回到最基本的：**把 Agent 当一个技术很强但不了解你项目背景的同事，然后像跟那个同事沟通一样去驱动它。**

你会发现，大部分低效协作的问题，其实是你还没想清楚自己要什么。
