# AI 协作实践：如何有效引导 Agent 发现与解决问题

本文档总结了在 GateFlow 项目中与 AI Coding Agent（Claude Code）协作的真实经验，提炼为一套可操作的问题引导方法。

## 核心原则

**Agent 的质量上限不由模型决定，而由你给出的上下文决定。** 同样的任务，提问方式从「帮我做 X」升级为「背景 + 约束 + 验证视角」，产出质量差距在 3 倍以上。

## 提问四要素

将复杂任务按以下结构组织，Agent 的初版方案命中率显著提高：

```
1. 背景（1 句）  →  为什么要做这件事？
2. 约束（1-2 条）→  什么绝对不能动？
3. 期望输出      →  分析？方案？执行？先审再做？
4. 审查视角      →  用什么角度验证结果？
```

### 对比示例

| 维度 | 低信息量 | 高信息量 |
|------|---------|---------|
| 任务陈述 | 「把 victor-stats 拆了」 | 「要把离线统计任务从在线服务拆出来单独部署」 |
| 约束 | 无 | 「不能动 victor-service 的 API；ClickHouse 连接配置保持一套」 |
| 期望输出 | 隐含（直接执行） | 「先分析现状，再给分阶段计划，我审完再执行」 |
| 审查视角 | 无 | 「执行完后，用『计算结果是否真正被写入』的视角再过一遍」 |

**低信息量版本**：Agent 会直接开始拆分模块，可能在方案设计阶段就与你的预期偏离，需要多轮修正。

**高信息量版本**：Agent 先探索代码结构、理解依赖关系、发现问题（双模式反模式），再提出分阶段方案。你在执行前就完成了方向校准。

## 三种有效的提问模式

### 模式 1：探索式诊断（先理解，后决策）

**不要**上来就下指令。用「为什么」开头，让 Agent 先做诊断。

**案例**：本次会话中，用户问「为什么 victor-service 会依赖 victor-stats 模块，victor-service 定位是什么？」这触发了 Agent 的架构分析，发现了双模式反模式（既是库又是独立进程），进而自然引出拆分方案。

**适用场景**：你不确定问题在哪，只知道有点不对劲。

### 模式 2：外部审查 + 内部验证

将其他 Agent 或工具的审查结果作为输入，但要求当前 Agent **先核实再执行**。

**案例**：外部 agent 提交了 16 个 issue 的审查报告。用户没有直接说「按这个改」，而是说「请检查核实，如果你认同，就尝试修复」。这避免了两个陷阱：

- 外部 agent 可能误判（如对某些「问题」的判断不符合实际代码逻辑）
- 当前 agent 不假思索地执行 todo list

**关键话术**：「请检查核实，如果你认同，就尝试修复。如果不认同，说明原因。」

### 模式 3：委托审查时指定视角

不要只说「帮我审查一下代码」，要指定审查维度。

**有效视角示例**：

| 视角 | 适用场景 |
|------|---------|
| 「追踪每个计算结果的消费者——如果计算结果只被 log 而未被任何字段使用，视为缺陷」 | 统计算法/数据管道 |
| 「检查 Java enum 和 MySQL ENUM 是否同步」 | 状态机变更 |
| 「对连续型指标是否误用了二项分布公式」 | 统计检验代码 |
| 「所有外部输入是否有空值和边界检查」 | 安全/健壮性审查 |

**案例**：本项目的统计管线审查中，外部 agent 追踪了 CUPED 计算结果的完整数据流，发现 `applyCUPED()` 的结果只被 debug 日志消费而从未写入报告——这是仅靠编译检查和功能测试无法暴露的逻辑缺陷。

## 常见误区

### 误区 1：把 Agent 当编译器

**错误**：「在 XXX 类里加一个 YYY 方法。」

**问题**：你已替 Agent 做了设计决策，Agent 只负责敲代码。但你可能还不知道有更好的方案。

**改进**：先说目的，让 Agent 提出方案。

### 误区 2：一次性给太多

**错误**：在一个 prompt 里塞 5 个独立任务。

**问题**：Agent 在任务间上下文污染，纠错成本高。

**改进**：一个 prompt 聚焦一个主题。相关的子任务可以放一起，但独立主题必须分开。

### 误区 3：不验证就说「好的」

**错误**：Agent 执行完后直接说「好的，下一步」。

**问题**：没有验收标准，Agent 不知道自己是否符合预期，下次无法校准。

**改进**：给一个简短的确认信号：「拆分方案确认，结构和我预期一致，继续」。或指出偏差：「scheduler 我希望用独立的 parent POM，不是继承 victor-parent」。

## 经验规则

- **应该**：复杂任务开头给出 1-2 条硬约束，不让 Agent 在不可能的方向上浪费探索时间。
- **应该**：用「先分析现状，再提方案，我审完再执行」控制节奏，避免一次性执行到底后才发现方向错误。
- **应该**：外部审查结果作为输入时，要求 Agent 先独立核实。不同 Agent 的信息视野不同，交叉验证才可靠。
- **禁止**：一次性粘贴大量外部报告并要求「全改」。Agent 会跳过逐项判断，直接批量执行。
- **禁止**：只给结论不给原因。「上次这么做有问题」没有信息量，要说「上次 mSPRT 用二项方差算连续型收入数据导致负数方差，这次别犯同类错误」。

## 沉淀与循环改进

每次会话结束后，用以下方式让下次协作更高效：

1. **知识库沉淀**（`/km-learn`）：将非显而易见的决策、坑、约束写入 `docs/knowledge/`，未来 Agent 可自动加载。
2. **验证规则固化**：将有效的审查视角（如「追踪计算结果消费者」）转为系统性的代码审查清单。
3. **CLAUDE.md 迭代**：将项目级约定（模块角色、命名规范、依赖策略）持续更新到 CLAUDE.md，Agent 每次启动自动加载。
