# GateFlow Tracker 数据采集系统设计

## 1. 概述

本文档描述 GateFlow Tracker（数据采集系统）的架构设计，目标是将用户行为数据采集与 A/B 实验系统分离，实现独立部署、独立扩展的数据采集平台。

### 1.1 设计目标

- **独立部署**：数据采集服务与 A/B 实验服务解耦，可独立运维
- **全面采集**：覆盖曝光、点击、停留时长、路径等维度
- **高性能**：支持高并发写入，支持批量上报
- **可扩展**：数据模型支持灵活扩展

## 3. 数据采集维度

### 3.1 采集指标矩阵

| 场景 | 核心指标 | 采集方式 | 说明 |
|------|----------|----------|------|
| **页面级分析** | PV/UV、跳出率、停留时长 | SPA 路由切换、visibilitychange API | 全局页面统计 |
| **元素级分析** | 曝光率、点击率、转化漏斗 | Intersection Observer | 关键元素追踪 |
| **路径分析** | 页面序列、关键节点、转化路径 | 会话级序列聚合 | 用户行为流 |
| **归因分析** | 各渠道贡献、首次/末次触点 | UTM 参数 + 时间窗口 | 渠道效果评估 |

### 3.2 曝光数据（Exposure）

| 字段 | 类型 | 说明 |
|------|------|------|
| 页面曝光 | Boolean | 页面/PV 级别标识 |
| 元素曝光 | Boolean | 关键元素进入可视区域 |
| 曝光时长阈值 | Number | 元素在视口停留时间（毫秒） |
| 曝光开始时间 | DateTime | 元素首次可见时间 |
| 曝光结束时间 | DateTime | 元素离开视口时间 |
| 有效曝光 | Boolean | 是否超过曝光阈值 |

### 3.3 点击数据（Click）

| 字段 | 类型 | 说明 |
|------|------|------|
| 点击事件 | Boolean | 可点击元素的点击行为 |
| 点击坐标 | {x, y} | 在元素上的相对位置 |
| 点击序列 | Number | 会话内点击顺序 |
| 元素标识 | String | data-id、元素路径 |
| 元素文本 | String | 按钮文字、链接文字 |
| 元素类型 | String | button、a、input 等 |

### 3.4 停留时长（Stay Duration）

| 字段 | 类型 | 说明 |
|------|------|------|
| 页面停留时长 | Number | 进入时间 - 离开时间（毫秒） |
| 元素停留时长 | Number | 用户在该区域的停留时间 |
| 有效阅读时长 | Number | 考虑滚动和视口的真实阅读时间 |
| 页面停留区间 | Enum | 25%, 50%, 75%, 100% |

### 3.5 页面交互深度

| 字段 | 类型 | 说明 |
|------|------|------|
| 滚动深度 | Number | 0-100% |
| 滚动速度 | Number | 像素/秒 |
| 停留区间 | Enum | 四分位停留 |
| 是否跳出 | Boolean | 单页会话标识 |

### 3.6 用户标识与上下文

| 字段 | 类型 | 说明 |
|------|------|------|
| userId | String | 用户唯一标识 |
| anonymousId | String | 匿名用户标识 |
| sessionId | String | 会话唯一标识 |
| 来源渠道 | String | utm_source |
| 营销媒介 | String | utm_medium |
| 营销活动 | String | utm_campaign |
| Referrer | String | 上一个页面地址 |
| 设备信息 | Object | UA、分辨率、操作系统 |

### 3.7 事件时序

| 字段 | 类型 | 说明 |
|------|------|------|
| timestamp | DateTime | 事件发生时间（毫秒） |
| eventType | String | page_view, click, exposure, custom |
| eventId | String | 事件唯一ID |
| elementId | String | 元素标识 |
| sequenceNumber | Number | 会话内事件序号 |

### 4.2 核心接口

```typescript
// Tracker 主类
class Tracker {
  constructor(config: TrackerConfig)

  // 初始化
  init(): Promise<void>

  // 手动采集
  track(event: EventData): void
  trackPageView(page: PageData): void
  trackClick(element: Element, data?: ClickData): void
  trackExposure(element: Element, data?: ExposureData): void

  // 批量上报
  flush(): Promise<void>

  // 会话管理
  startSession(): void
  endSession(): void

  // 销毁
  destroy(): void
}

// 事件类型
type EventType =
  | 'page_view'      // 页面浏览
  | 'click'          // 点击
  | 'exposure'       // 元素曝光
  | 'scroll'         // 滚动
  | 'stay'           // 停留
  | 'custom'         // 自定义
```

### 4.3 自动采集配置

```typescript
interface AutoTrackConfig {
  // 页面浏览

    enabled: boolean
   SPA: boolean                    // 是否监听 SPA 路由变化

  }

  // 点击采集
  click: {
    enabled: boolean
    selector: string[]              // 采集的选择器
    excludeSelector: string[]      // 排除的选择器
    trackText: boolean             // 是否采集元素文本
    trackPosition: boolean          // 是否采集点击坐标
  }

  // 曝光采集
  exposure: {
    enabled: boolean
    selector: string[]             // 曝光监测的选择器
    threshold: number              // 曝光阈值（毫秒）
    thresholdRatio: number         // 曝光比例阈值（0-1）
    observerOptions: IntersectionObserverInit
  }

  // 滚动采集
  scroll: {
    enabled: boolean
    thresholds: number[]            // 采集深度阈值 [25, 50, 75, 100]
    throttle: number               // 节流间隔
  }

  // 停留采集
  stay: {
    enabled: boolean
    threshold: number               // 有效停留最小时间
  }
}
```

## 6. 数据模型

### 6.1 事件表 (events)

```sql
CREATE DATABASE IF NOT EXISTS gateflow_tracker;

CREATE TABLE IF NOT EXISTS gateflow_tracker.events (
    -- 基础字段

    -- 页面上下文
    page_url       String,
    page_title     String,
    page_referrer  String,

    -- 会话信息
    session_start  DateTime64(3),

    -- 设备信息
    device_type    String,        -- mobile, desktop, tablet
    os             String,
    browser        String,
    screen_width   UInt32,
    screen_height  UInt32,
    language       String,

    -- 交互数据
    element_id     String,
    element_type   String,
    element_text   String,
    click_x        Nullable(Int32),
    click_y        Nullable(Int32),
    scroll_depth   Nullable(UInt8),
    stay_duration  Nullable(Int64),

    -- 归因数据
    utm_source     String,
    utm_medium     String,
    utm_campaign   String,
    utm_term       String,
    utm_content    String,

    -- 实验关联（可选）
    exp_ids        Array(String),
    variants       Array(String),

    -- 自定义属性 (JSON)
    properties     String,

    -- 元数据
    event_index    UInt64
) ENGINE = MergeTree()
PARTITION BY toYYYYMMDD(timestamp)
-- 优化：user_id 置顶，用户中心化查询更高效
ORDER BY (user_id, timestamp, event_type, session_id)
-- 数据保留策略：90 天自动清理
TTL timestamp + INTERVAL 90 DAY
SETTINGS index_granularity = 8192;

-- 索引
ALTER TABLE events ADD INDEX idx_user_id user_id TYPE bloom_filter(0.01) GRANULARITY 4;
ALTER TABLE events ADD INDEX idx_session session_id TYPE bloom_filter(0.01) GRANULARITY 4;
ALTER TABLE events ADD INDEX idx_event_type event_type TYPE set(100) GRANULARITY 4;
ALTER TABLE events ADD INDEX idx_page_url page_url TYPE bloom_filter(0.01) GRANULARITY 4;
```

## 7. 与 A/B 系统关联

### 7.1 关联方式

通过 `user_id` 实现事件数据与实验数据的关联：

```sql
-- 查询实验组用户的行为数据
SELECT
    e.variant,
    COUNT(DISTINCT t.user_id) as users,
    COUNT(DISTINCT t.session_id) as sessions,
    AVG(t.stay_duration) as avg_stay,
    COUNT(t.event_id) as events
FROM victor_experiments e
JOIN victor_user_assignments a ON e.id = a.experiment_id
JOIN gateflow_tracker.events t ON a.user_id = t.user_id
WHERE e.experiment_key = 'homepage_banner_v2'
  AND t.timestamp BETWEEN a.assigned_at AND a.assigned_at + INTERVAL 7 DAY
GROUP BY e.variant;
```

## 9. 可靠性设计

### 9.1 Dead Letter Queue (DLQ)

事件处理失败时（如验证失败、写入失败、熔断触发），将事件写入 DLQ 供后续重放：

```
失败事件 → DLQ (Redis, 7天TTL) → 定时任务重放 → ClickHouse
```

**失败原因分类**：
- `validation_failed` — 事件格式校验不通过
- `circuit_breaker_open` — ClickHouse 熔断器打开
- `clickhouse_write_failed` — ClickHouse 写入异常
- `enrichment_failed` — 数据增强失败（较少见）

### 9.2 熔断机制

使用 Resilience4j 实现 ClickHouse 写入熔断：

```
正常：写入成功率 > 50% → 熔断器关闭
异常：连续失败率达阈值 → 熔断器打开（30秒）
恢复：30秒后进入半开状态，尝试放行少量请求
```

熔断打开时，新事件直接写入 DLQ，不阻塞主流程。

### 9.3 数据保留策略

通过 ClickHouse TTL 实现自动数据清理：

```sql
-- 事件表：90 天保留
ALTER TABLE gateflow_tracker.events MODIFY TTL timestamp + INTERVAL 90 DAY;

-- 会话表：90 天保留
ALTER TABLE gateflow_tracker.sessions MODIFY TTL start_time + INTERVAL 90 DAY;
```

### 9.4 物化视图预聚合

高频查询使用物化视图预计算：

| 视图 | 粒度 | 用途 |
|------|------|------|
| `events_hourly` | 小时级 | 实时仪表盘 |
| `sessions_daily` | 日级 | 日报表生成 |

## 10. 可扩展性设计

### 10.1 Redis Cluster 兼容去重

单实例 Redis 在高吞吐下会成为瓶颈。Redis Cluster 环境下使用哈希槽分片：

```java
// 使用 eventId 的 CRC16 哈希决定槽位
int slot = crc16(eventId) % 16384;
String key = "dedup:" + eventId;
// Redis Cluster 自动将 key 路由到对应槽位的节点
```

### 10.2 ClickHouse 集群

生产环境建议部署 ClickHouse 集群：

```
┌─────────────────┐       ┌─────────────────┐

                          ┌────────▼────────┐
                          │ ClickHouse      │
                          │ Replica 2       │
                          │ (查询)          │
                          └─────────────────┘
```
