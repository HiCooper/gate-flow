# AI 编程协作实战：提问的质量，决定了代码的质量

> 同样的模型、同样的工具，有人用出 10 倍效率，有人觉得鸡肋。模型是基座，决定能力下限；但真正拉开差距的，是上下文。

过去半年在一个大型 A/B 实验平台项目中深度使用 AI Coding Agent。模型选型锁定之后，仅靠改进提问方式，产出质量差距就在 3 倍以上。以下是踩过的坑和验证有效的经验。

---

## 一、模型决定下限，上下文决定上限

**在模型能力一致的前提下，Agent 的输出质量上限由上下文决定。**

想象你带一个技术很强但完全不了解项目的实习生。你只说「把这个模块拆了」，他会按自己理解直接上手，方向偏了需要多轮纠错。

但如果你说：

> 离线计算任务耦合在在线服务里导致发布互相阻塞，要拆出去单独部署。约束：不能动对外 API，数据库连接保持一套。你先分析现状，提分阶段方案，我确认后再执行。

同样的实习生，同样的起点——产出质量截然不同。

---

## 二、提问四要素

经过数十次协作迭代，提炼出一个好 prompt 的结构：

```text
1. 背景（1 句）  →  为什么做？
2. 约束（1-2 条）→  什么不能动？
3. 期望输出      →  分析 / 方案 / 执行？先审再做？
4. 审查视角      →  用什么标准验证？
```

### 对比

| 维度 | 低信息量 | 高信息量 |
|------|---------|---------|
| 任务 | 「把统计模块拆了」 | 「离线统计从在线服务拆出，独立部署」 |
| 约束 | 无 | 「不动业务 API，数据库连接保持一套」 |
| 输出 | 隐含「直接执行」 | 「先分析现状，再给分阶段计划，审完执行」 |
| 审查 | 无 | 「执行后以『计算结果是否真正写入』视角再过一遍」 |

- **低信息量**：Agent 直接拆，方案阶段就可能偏离预期，多轮修正，且越偏越远。
- **高信息量**：Agent 先探索代码结构、理解依赖、主动发现架构反模式（模块既是库又是独立进程），再提方案。动手前已完成方向校准。

一个 prompt 多花 30 秒，省掉后续 30 分钟的纠错成本。

## 三、三种高杠杆提问模式

### 模式一：探索式诊断

**不需要知道答案，但需要问对问题。**

隐约觉得不对劲又说不清时，用「为什么」开头，而非直接下指令。

**案例**：感觉服务依赖关系奇怪，问了一句：

> 为什么业务服务模块会依赖统计计算模块？业务服务的定位是什么？

Agent 由此触发架构分析，跑完代码后发现了双模式反模式——同一模块既要作为库被引用，又要作为独立进程部署，两者生命周期、配置、依赖方向完全不同。

如果当初说的是「帮我把统计相关引用理一下」，只会得到一张引用表，架构问题永远不会暴露。

**适用场景**：感到不对劲但说不清问题在哪——这正是 AI 诊断的强项。

---

### 模式二：外部审查 + 独立验证

直接扔给 Agent 一个 issue 列表说「全改了」，它会当成待办清单逐项执行，不会独立判断合理性。

**正确做法**：

> 请独立核实这些审查结论。认同的修复，不认同的说明原因。

**实测对比**：外部审查 16 个问题。不加核实指令，采纳率 100%（照单全收）。加核实指令后，Agent 自行推翻了 4 个——审查者对业务逻辑理解有误、上下文版本已过期。

**关键话术**：先核实，再执行。不同意的给出原因。

---

### 模式三：指定审查视角

「帮我审查代码」不指定视角，Agent 只看表层——命名、格式、明显的空指针。

**验证有效的审查视角**：

```text
追踪每个计算结果的消费者
→ 如果结果只被日志打印而未被任何字段使用，视为缺陷

检查枚举定义和数据字典是否同步
→ 状态机变更

连续型指标是否误用了离散分布公式
→ 统计检验代码

所有外部输入是否有空值和边界检查
→ 健壮性审查
```

**案例**：在统计管线审查中，要求 Agent 追踪方差缩减算法每个计算结果：

> 这个函数输出什么？谁消费了它？最终有没有写入报告？

Agent 按此视角跑完后发现：核心函数 `apply()` 的结果只被一行 debug 日志消费，从未进入任何报告字段。

编译检查发现不了，单测覆盖不了，功能测试测不出来。只有「追踪数据流最终消费」这个视角才能暴露。

## 四、三个常见误区

### 误区一：把 Agent 当编译器

**症状**：「在 XX 类加一个 YY 方法。」

你已替 Agent 做了技术决策，它只是敲键盘。但 YY 方法真的是最优解吗？

**改进**：说目的，不说实现。把设计空间留给 Agent。

```diff
- 给 UserService 加一个 getUserStats 方法
+ 管理员面板需要用户聚合统计数据。看现有代码结构，给个实现方案。
```

---

### 误区二：一个 prompt 塞太多东西

**症状**：一段话塞 5 个独立任务。

Agent 顺序处理信息，任务 A 的上下文会污染任务 B，任务 C 的错误会被带到任务 D。

**改进**：一个 prompt 一个主题。多开几个会话窗口的成本，远低于在一个窗口里反复纠错。

---

### 误区三：不给反馈信号

**症状**：Agent 执行完你说「好的，下一步」。

「好的」是零信息——Agent 不知道输出是否符合预期，下次无法校准。

**改进**：关键决策点给短反馈。

```diff
- 好的，下一步
+ 拆分方案确认，结构和我预期一致，继续。
+ 调度模块用独立父 POM，不继承统一父 POM，其他没问题。
```

## 五、经验规则

1. **复杂任务开头给硬约束。** 一两句话圈定禁区（不能改的 API、不能动的配置、不能引入的新依赖），Agent 直接在有效空间内搜索最优解。

2. **用「先分析 → 提方案 → 审完执行」控制节奏。** 投入产出比最高的话术。多花 1 分钟对齐方案，可能节省 30 分钟返工。

3. **外部输入必须独立核实。** 不同 Agent 信息视野不同，交叉验证是工程常识，不是不信任。

4. **给结论的同时给原因。**「上次这么做有问题」没信息量。要说「上次序贯检验对连续型收入数据误用了二项分布公式导致方差为负，这次碰到同类统计代码先确认分布假设。」

---

## 六、持续改进

每次协作结束后花 2 分钟：

1. **沉淀决策与约束。** 非显而易见的坑、架构约束、命名约定写入知识库，后续 Agent 自动加载。

2. **固化审查视角。** 验证有效的审查维度沉淀为团队 checklist。一个好视角可以反复复用。

3. **维护项目上下文。** 持续更新模块角色、依赖策略、命名规范——Agent 每次启动自动加载，相当于每次都有「老员工」级别的上下文。

---

## 写在最后

AI 编程助手进步很快，但有一个东西不变：**工具的能力上限是你的表达上限。** 你能多清晰描述背景、约束和验收标准，它就能多好地帮你解决问题。

与其研究 prompt 模板和技巧，不如回到最基本的：

**把 Agent 当一个技术很强但不了解项目背景的同事，然后像跟那个同事沟通一样去驱动它。**

大部分低效协作，其实源于你还没想清楚自己要什么。
