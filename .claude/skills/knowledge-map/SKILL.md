# Knowledge Map — 项目知识库导航与维护

本 Skill 管理 `docs/knowledge/` 下的增量式知识库，专为 AI Coding Agent 设计。知识库按主题分目录组织，支持按需渐进加载。

## 目录速查

```
docs/knowledge/
├── 01-project-overview/     # 项目目标、高层架构、术语表
├── 02-business-domain/      # 业务规则、业务术语、运营逻辑
├── 03-development-guide/    # 环境搭建、编码规范、项目约定
├── 04-best-practices/       # 已验证模式、性能技巧、设计准则
├── 05-historical-lessons/   # 事故复盘、踩坑记录、已废弃决策
├── 06-api-reference/        # API 契约、接入说明、外部服务文档
├── 07-test-rule/            # 测试规范、用例设计、质量门禁规则
└── 08-externel-resources/   # 外部链接、参考资料、第三方文档
```

## 命令列表

### /km-init — 初始化知识库结构

**用途：** 在新项目或新分支上初始化知识库目录骨架和 README 索引。

**执行步骤：**

1. 检查 `docs/knowledge/` 是否已存在。如已存在，报告当前结构，不做重复创建。
2. 如不存在，创建以下目录骨架：
   ```
   docs/knowledge/
   ├── 01-project-overview/
   ├── 02-business-domain/
   ├── 03-development-guide/
   ├── 04-best-practices/
   ├── 05-historical-lessons/
   ├── 06-api-reference/
   ├── 07-test-rule/
   └── 08-externel-resources/
   ```
3. 生成 `docs/knowledge/README.md`，包含目录索引表（Catalog）和文档格式模板。Catalog 表初始为空。
4. 从 CLAUDE.md、项目 README、已有 docs 文件中提取关键信息，生成首批知识文档的草稿建议列表（不自动写入，仅列出建议）。
5. 报告初始化结果：已创建的目录数、README 状态、建议的首批文档清单。

### /km-update <path> — 分析代码并更新文档

**用途：** 分析指定代码路径的变更，更新知识库中对应的文档。

**执行步骤：**

1. 读取 `<path>` 下的代码文件，理解其功能、接口、约束。
2. 确定该代码属于哪个知识主题（参考目录速查表），定位最匹配的子目录。
3. 扫描该子目录下的现有 `.md` 文件，检查是否有与该代码相关的已有文档。
4. 如有已有文档：对比代码现状与文档内容，更新过时部分，补充新增约束或接口变化。
5. 如无已有文档：创建新文档，按以下格式模板撰写，使用 kebab-case 文件名。
6. 更新 `docs/knowledge/README.md` 中的 Catalog 表，注册新文档或标注文档已更新。
7. 输出变更摘要：哪些文档被更新、哪些被新建、哪些知识点被补充。

**文档格式模板：**

```markdown
# 标题

**类型：** [决策 | 规范 | 业务规则 | 排坑记录 | 操作指南 | 参考]
**范围：** [前端 | 后端 | 基础设施 | 全局 | 模块名]
**前置依赖：** [无 | 路径/到/依赖文档.md]
**最后更新：** YYYY-MM-DD

## 摘要（TL;DR）

1-2 句话概括核心结论。

## 背景

本文档要解决什么问题？

## 详情

核心知识。优先 bullet points 和具体示例。

## 可执行规则

- **应该：** ...
- **禁止：** ...

## 关联文档

- [主题名](路径/到/文件.md) — 关联原因
- `src/路径/到/代码` — 实现该知识的代码位置
```

### /km-learn <topic> — 记录经验 / 捕获会话

**用途：** 将当前会话中与 `<topic>` 相关的经验、决策或发现捕获到知识库。

**执行步骤：**

1. 回顾当前会话对话历史，提取与 `<topic>` 相关的关键信息：
   - 做了什么决策？为什么？
   - 遇到了什么问题？怎么解决的？
   - 发现了什么非显而易见的约束或规则？
   - 有什么值得下次避免的坑？
2. 确定该主题属于哪个知识子目录（参考目录速查表）。
3. 检查该子目录下是否已有同名或相近主题的文档。如有，合并补充；如无，新建。
4. 撰写文档内容，严格遵循文档格式模板。特别注意：
   - 「可执行规则」必须是从经验中提炼的显式 Do/Don't 规则。
   - 「背景」必须说明这个经验是在什么场景下获得的。
5. 更新 Catalog 表。
6. 输出捕获摘要：文档路径、新增知识点数、合并知识点数。

### /km-summary — 会话总结与知识沉淀

**用途：** 在会话结束前，扫描整个会话历史，将所有有价值的知识沉淀到知识库。

**执行步骤：**

1. 扫描当前会话完整对话历史，识别以下类别的知识：
   - **决策类** — 做了什么技术/业务决策，为什么，有什么后果。
   - **发现类** — 发现了什么非显而易见的代码行为、隐藏约束、边界情况。
   - **排坑类** — 遇到了什么 bug 或陷阱，根因是什么，怎么修的。
   - **模式类** — 验证了什么编码模式或设计模式有效。
   - **规范类** — 确立了什么新的约定或标准。
2. 对每条知识，按主题分类到对应子目录。
3. 检查已有文档是否存在重叠或可合并。优先合并到已有文档，而非创建新文件。
4. 撰写/更新文档，遵循格式模板。
5. 更新 Catalog 表。
6. 输出沉淀报告：
   - 新建文档数 + 路径列表
   - 更新文档数 + 路径列表
   - 每个文档的核心知识点摘要（1 行）
   - 未沉淀的会话内容说明（如有，解释为什么不值得沉淀）

### /km-clean — 清理维护

**用途：** 检测知识库中的过期、重复、低质量内容，并自动改进。

**执行步骤：**

1. **过期检测：**
   - 读取每篇文档的「最后更新」日期。
   - 对超过 30 天未更新的文档，验证其引用的代码路径是否仍存在（用 Glob/Grep 检查）。
   - 如代码已不存在，标记文档为「已过期」，在文档顶部添加废弃声明块：
     ```markdown
     > **已过期（YYYY-MM-DD）**：本文档引用的代码已不存在或逻辑已变更。请以代码为准。
     ```
   - 如代码仍存在但逻辑可能已变，在文档顶部添加「待验证」标记。
2. **重复检测：**
   - 扫描所有文档的标题和摘要，识别主题重叠的文档。
   - 对重叠文档，建议合并方案：保留哪篇、合并哪些内容、删除哪篇。
   - 执行合并（需用户确认），合并后在被删除的文档位置放置重定向：
     ```markdown
     > **内容已合并至** [新文档名](路径/到/新文档.md)
     ```
3. **低质量检测：**
   - 检查每篇文档是否包含所有必需字段（类型、范围、摘要、可执行规则）。
   - 对缺失字段的文档，自动补充（从代码和上下文推断）。
4. **Catalog 校验：**
   - 验证 Catalog 表中的每条路径对应的文件是否实际存在。
   - 验证知识库中每个 `.md` 文件是否在 Catalog 中有注册。
   - 修复不一致：删除 Catalog 中指向不存在文件的条目，为未注册文件添加条目。
5. 输出清理报告：过期文档数、合并文档数、补充字段数、Catalog 修复数。

### /km-eval — 会话评估

**用途：** 6 维度评估当前会话的知识利用情况，并给出自动反馈。

**评估维度：**

1. **知识加载率** — 本次会话是否按需加载了相关知识？加载了多少篇？是否遗漏了应该加载的？
2. **知识应用率** — 加载的知识中，「可执行规则」是否被实际应用到代码/决策中？应用比例？
3. **知识沉淀率** — 会话中产生的有价值知识，有多少被沉淀到知识库？遗漏了什么？
4. **知识冲突率** — 知识文档与代码现状是否存在矛盾？有多少冲突被发现？
5. **知识覆盖率** — 当前知识库对项目关键领域的覆盖程度？哪些领域还是空白？
6. **知识质量分** — 文档格式规范性（必需字段完整度）、内容准确性（代码引用是否有效）、时效性（更新日期是否近期）。

**执行步骤：**

1. 回顾当前会话历史，按上述 6 个维度逐项评估。
2. 对每个维度，给出 1-5 分评分 + 具体说明。
3. 计算综合评分（6 维度加权平均）。
4. 生成自动反馈：
   - **改进建议** — 针对得分最低的 2 个维度，给出具体可执行的改进动作。
   - **待沉淀清单** — 会话中尚未沉淀但值得沉淀的知识点列表。
   - **待补充清单** — 知识库中缺失但本次会话暴露出需要的领域。
5. 输出评估报告表格 + 改进建议。

## 通用原则

- **按需加载** — 禁止一次性读取整个知识库。先读 README Catalog，按关键词匹配，再定向加载。
- **原子化** — 一篇文档只讲一个主题，保持小而聚焦。
- **可链接** — 用「前置依赖」和「关联文档」构建知识网络。
- **带日期** — 必须填写「最后更新」，Agent 优先信任近期文档。
- **代码为准** — 当知识文档与代码现状矛盾时，以代码为准，同时标记文档待验证。
